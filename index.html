📖 Giới thiệu

Skill là gì và vì sao bạn cần nó?

Hiểu rõ khái niệm skill — công cụ giúp AI Assistant hoạt động nhất quán, thông minh hơn và giảm thiểu thao tác lặp lại của bạn.

Trong bối cảnh AI Assistant ngày càng phổ biến, một trong những thách thức lớn nhất là làm sao để trợ lý ảo hiểu đúng ý bạn mà không cần giải thích lại mỗi lần. Skill chính là giải pháp: đó là một bộ hướng dẫn được đóng gói cẩn thận, chứa đầy đủ thông tin về quy trình, ngữ cảnh và ràng buộc để AI có thể thực thi một nhiệm vụ cụ thể một cách chính xác, nhất quán.

Hãy hình dung skill giống như một cuốn sổ tay tác nghiệp — nhưng dành cho AI. Thay vì bạn phải viết prompt dài dòng mỗi lần cần AI làm một việc gì đó, skill cho phép bạn chuẩn hóa kiến thức chuyên môn thành tài liệu có cấu trúc, để AI tự biết cách thực hiện đúng khi gặp tình huống phù hợp.

Khi nào nên dùng skill?

  • Bạn thường xuyên viết prompt tương tự cho cùng một loại tác vụ
  • Bạn muốn AI tuân thủ quy trình cố định mà không cần nhắc nhở
  • Team của bạn cần chuẩn hóa cách AI xử lý công việc nội bộ
  • Bạn muốn chia sẻ kiến thức chuyên gia cho người khác dưới dạng có thể tái sử dụng

Lợi ích khi sử dụng skill

⚡
Tiết kiệm thời gian
Không cần viết prompt lặp lại. Skill tự kích hoạt khi AI nhận diện đúng ngữ cảnh.
🎯
Kết quả nhất quán
Mỗi lần thực thi đều tuân theo cùng quy trình, giảm sai sót và thiếu nhất quán.
🔄
Dễ chia sẻ & tái sử dụng
Skill là tài sản kiến thức — bạn có thể chia sẻ cho team hoặc cộng đồng.
🧩
Linh hoạt mở rộng
Kết hợp nhiều skill hoặc tích hợp với tool, connector, MCP để tạo workflow mạnh mẽ hơn.
"Skill là cách bạn đóng gói hướng dẫn, workflow và kiến thức chuyên môn để AI sử dụng đúng lúc, đúng cách — không cần bạn lặp lại." — Triết lý thiết kế skill

Nền Tảng Cơ Bản

Hiểu rõ bản chất của skill, các thành phần cốt lõi và nguyên tắc thiết kế nền tảng.

Skill trong AI Assistant là gì?

Skill là một đơn vị kiến thức được đóng gói theo cấu trúc chuẩn, chứa đầy đủ hướng dẫn để AI Assistant thực hiện một tác vụ hoặc nhóm tác vụ liên quan. Mỗi skill hoạt động như một "module chuyên gia" — khi được kích hoạt, AI sẽ tuân theo quy trình đã định nghĩa thay vì phải suy luận từ đầu.

Khác với prompt đơn lẻ, skill có tính bền vững (dùng lại nhiều lần), cấu trúc rõ ràng (file, thư mục, metadata) và khả năng mở rộng (kết hợp với tool, API, MCP).

Thành phần cốt lõi của một skill

📁 Cấu trúc thư mục skill
your-skill-name/
├── SKILL.md          # File hướng dẫn chính (bắt buộc)
├── scripts/          # Kịch bản hỗ trợ, helper
├── references/       # Tài liệu tham chiếu bổ sung
└── assets/           # Template, config, tài nguyên

SKILL.md — Trái tim của mỗi skill

File SKILL.md chứa toàn bộ hướng dẫn mà AI cần để thực hiện tác vụ. File này gồm 2 phần chính: frontmatter (metadata YAML) và nội dung markdown (hướng dẫn chi tiết). AI sẽ đọc frontmatter trước để quyết định có nên kích hoạt skill hay không, sau đó mới đọc nội dung chi tiết.

scripts/ — Kịch bản hỗ trợ

Thư mục này chứa các script mà skill có thể gọi trong quá trình thực thi, ví dụ: script deploy, script validation, script generate code.

references/ — Tài liệu mở rộng

Khi hướng dẫn quá dài hoặc cần tham chiếu chuyên sâu, bạn tách ra file riêng và đặt trong thư mục này. AI sẽ đọc khi cần chi tiết hơn.

Nguyên tắc Progressive Disclosure

Skill được thiết kế theo nguyên tắc "tiết lộ dần" — AI chỉ đọc những gì cần thiết ở mỗi thời điểm, tránh quá tải:

🏷️
Tầng 1: Frontmatter
AI đọc metadata (tên, mô tả) để quyết định có kích hoạt skill hay không. Nhẹ và nhanh.
📄
Tầng 2: Nội dung chính
Khi skill được kích hoạt, AI đọc toàn bộ hướng dẫn trong SKILL.md để thực thi.
📚
Tầng 3: File tham chiếu
Chỉ khi cần chi tiết chuyên sâu, AI mới đọc thêm các file trong references/ hoặc scripts/.

Composability — Khả năng ghép nối

Một skill tốt có thể kết hợp với skill khác hoặc với công cụ bên ngoài. Ví dụ: skill "tạo component React" có thể gọi skill "viết unit test" và kết nối với MCP server để đẩy code lên repository. Thiết kế skill theo hướng module nhỏ, rõ ràng sẽ giúp bạn tái sử dụng linh hoạt.

Portability — Tính di động

Skill được viết dưới dạng file markdown và thư mục đơn giản — không phụ thuộc platform cụ thể. Bạn có thể chuyển skill giữa các dự án, share qua Git, hoặc đóng gói để cộng đồng sử dụng.

💡 Ghi nhớ nhanh
Skill = SKILL.md + scripts + references + assets. AI đọc theo 3 tầng (frontmatter → nội dung → tham chiếu). Thiết kế skill nhỏ, rõ ràng, dễ ghép nối và không phụ thuộc platform.

Lập Kế Hoạch & Thiết Kế

Mọi skill hiệu quả đều bắt đầu từ một use case rõ ràng và quy trình được mô tả cẩn thận.

Bắt đầu từ use case

Đừng vội viết skill. Hãy bắt đầu bằng câu hỏi: "Tác vụ cụ thể nào tôi muốn AI thực hiện?". Mô tả rõ tình huống người dùng gặp phải, hành động mong muốn và kết quả cuối cùng. Use case càng cụ thể, skill càng chính xác.

Xác định trigger

Trigger là điều kiện để AI nhận biết "đã đến lúc dùng skill này". Trigger có thể dựa trên: từ khóa trong câu hỏi của user, loại file đang mở, ngữ cảnh dự án, hoặc lệnh trực tiếp từ user.

Mô tả workflow nhiều bước

Phần lớn skill thực tế đòi hỏi AI thực hiện nhiều bước tuần tự. Hãy mô tả từng bước rõ ràng: bước nào làm trước, đầu ra của bước này là đầu vào bước tiếp theo, điều kiện dừng hoặc xử lý lỗi.

Ví dụ minh họa: Skill tạo API endpoint

📋 Ví dụ mô tả use case
Use case:   Tạo API endpoint RESTful
Trigger:    User yêu cầu tạo endpoint mới
Input:      Tên resource, các field, quan hệ
Output:     File controller, service, model, route

Workflow:
  1. Phân tích yêu cầu → xác định entity
  2. Tạo model/schema
  3. Tạo service layer
  4. Tạo controller + route
  5. Viết validation & error handling
  6. Tạo test cơ bản

Ba nhóm use case phổ biến

Nhóm Mô tả Ví dụ
Tạo tài liệu/asset Skill sinh ra file, code, document theo template Tạo component, viết docs, generate config
Tự động hóa workflow Skill thực hiện quy trình nhiều bước tự động CI/CD pipeline, review code, deploy
Tích hợp tool/MCP Skill gọi tool bên ngoài để lấy hoặc đẩy dữ liệu Query database, push Jira ticket, gọi API

Checklist: Trước khi viết skill

✓
Đã xác định rõ use case cụ thể
✓
Đã mô tả trigger kích hoạt
✓
Đã xác định input / output rõ ràng
✓
Đã phác thảo workflow từng bước
✓
Đã xác định tool/constraint cần thiết
✓
Đã xem xét edge case và error handling

Tiêu Chí Thành Công & Cách Đánh Giá

Đo lường hiệu quả skill bằng cả chỉ số định lượng lẫn định tính — không chỉ "chạy được" mà phải "chạy tốt".

Metric định lượng

⏱️
Thời gian hoàn thành
So sánh thời gian AI hoàn thành tác vụ có skill vs. không có skill. Mục tiêu: giảm ≥50%.
🎯
Tỷ lệ chính xác
Phần trăm output đúng theo yêu cầu ngay lần đầu, không cần sửa hoặc retry.
🔄
Số lần retry
Skill tốt giúp giảm số lần user phải yêu cầu lại hoặc chỉnh sửa output.
📏
Độ nhất quán
Chạy cùng input nhiều lần — output có giống nhau về cấu trúc và chất lượng không?

Metric định tính

  • Dễ hiểu: Người mới có thể đọc skill và hiểu workflow trong 5 phút không?
  • Dễ sửa: Khi cần thay đổi logic, có thể chỉnh sửa nhanh mà không phá vỡ cấu trúc?
  • Dễ mở rộng: Có thể thêm bước mới hoặc tích hợp tool mà không cần viết lại từ đầu?

So sánh: Có skill vs. Không có skill

Tiêu chí Không có Skill Có Skill
Thời gian prompt Viết lại mỗi lần, 3-5 phút Tự kích hoạt, <10 giây
Nhất quán output Thay đổi theo cách viết prompt Chuẩn hóa theo workflow
Chia sẻ kiến thức Khó — phụ thuộc cá nhân Dễ — share skill file
Bảo trì Sửa prompt rải rác Sửa 1 file, áp dụng toàn bộ
Khả năng mở rộng Hạn chế Kết hợp tool, MCP, skill khác
💡 Mẹo đánh giá
Tạo một bộ 5-10 câu test query đa dạng. Chạy skill với từng query và chấm điểm output theo thang 1-5. Skill đạt yêu cầu khi trung bình ≥ 4/5.

Yêu Cầu Kỹ Thuật & Cấu Trúc Thư Mục

Tuân thủ quy tắc đặt tên và cấu trúc chuẩn để skill hoạt động ổn định trên mọi môi trường.

Cấu trúc thư mục chuẩn

📁 Cấu trúc bắt buộc
your-skill-name/              # kebab-case, viết thường
├── SKILL.md                  # Bắt buộc — viết hoa đúng
├── scripts/                  # Tùy chọn
│   ├── generate.sh
│   └── validate.py
├── references/               # Tùy chọn
│   ├── api-spec.md
│   └── best-practices.md
└── assets/                   # Tùy chọn
    ├── template.yaml
    └── example-output.md

Quy tắc đặt tên

✓
File chính phải đặt tên là SKILL.md (viết hoa, không đổi tên)
✓
Tên thư mục dùng kebab-case (VD: api-generator, code-reviewer)
✓
Không dùng README.md thay cho SKILL.md trong thư mục skill
✓
Script và reference file cũng dùng kebab-case

Lỗi phổ biến cần tránh

⚠️ Cảnh báo
1. Đặt tên file sai: Viết skill.md hoặc Skill.md — AI sẽ không nhận diện được.
2. Thiếu frontmatter: Không có metadata YAML → AI không biết khi nào nên kích hoạt skill.
3. Nội dung quá dài: Nhồi mọi thứ vào 1 file → AI bị quá tải. Hãy tách ra references/.
4. Không có ví dụ: Hướng dẫn thiếu ví dụ cụ thể → AI hiểu sai ý định.

Patterns Thực Chiến

Các mẫu thiết kế đã được kiểm chứng — áp dụng ngay cho skill của bạn.

Pattern 1: Standalone Skill

Dành cho tác vụ đơn lẻ, không cần gọi tool bên ngoài. Ví dụ: tạo component, viết documentation, review code theo convention.

📋 Template — Standalone Skill
---
name: ten-skill-cua-ban
description: "Mô tả ngắn gọn skill làm gì"
---

# Tên Skill
Mô tả vai trò và mục đích của skill.

## Khi nào kích hoạt
Liệt kê các trigger cụ thể.

## Quy trình thực hiện
1. Bước 1: ...
2. Bước 2: ...
3. Bước 3: ...

## Ví dụ đầu ra
Cung cấp mẫu output mong muốn.

Pattern 2: Workflow Automation

Dành cho quy trình nhiều bước tuần tự. Mỗi bước có input/output rõ ràng, có điều kiện kiểm tra và xử lý lỗi.

Pattern 3: Skill + Tool/MCP

Dành cho skill cần tương tác với hệ thống bên ngoài (database, API, CI/CD). Skill mô tả workflow, tool cung cấp khả năng thực thi.

Best practices

  • Luôn bắt đầu bằng mô tả rõ trigger — khi nào AI nên dùng skill này
  • Từng bước trong workflow phải có hành động cụ thể, không mô tả mơ hồ
  • Thêm phần validation sau mỗi bước quan trọng
  • Định nghĩa fallback rõ ràng: nếu step X thất bại thì làm gì?
  • Giữ skill nhỏ gọn, tập trung 1 mục đích — dễ tái sử dụng hơn skill "làm mọi thứ"

Kiểm Thử, Cải Tiến & Phân Phối

Từ bản nháp đến sản phẩm hoàn chỉnh — quy trình biến skill thô thành skill chất lượng cao.

Quy trình từ Draft đến Publish

1
Draft
Viết bản nháp đầu tiên dựa trên use case và template. Chưa cần hoàn hảo.
2
Test
Tạo 5-10 câu test query đa dạng. Chạy skill và đánh giá output theo các tiêu chí.
3
Refine
Sửa wording, thêm ví dụ, bổ sung edge case dựa trên kết quả test.
4
Review
Nhờ đồng nghiệp hoặc AI khác review skill. Thu thập phản hồi.
5
Publish
Đóng gói skill hoàn chỉnh, viết changelog và chia sẻ cho team/cộng đồng.

Cách tạo bộ test query

Bộ test nên bao gồm: trường hợp điển hình (happy path), trường hợp biên (edge case), input sai định dạng, và yêu cầu mơ hồ. Mục tiêu: skill phải xử lý đúng ≥80% test query.

Cách chia sẻ skill

  • Nội bộ team: Đặt skill vào folder chung hoặc Git repository của team
  • Cộng đồng: Đóng gói theo chuẩn, thêm README, ví dụ và LICENSE rồi publish lên GitHub
  • Tài liệu đi kèm: Mỗi skill chia sẻ nên có: mô tả use case, hướng dẫn cài đặt, ví dụ sử dụng, và changelog

# React Component Generator
Bạn là chuyên gia tạo React component chuẩn production.

## Khi nào kích hoạt
- User yêu cầu tạo component mới
- User cần thêm UI element vào dự án React

## Quy trình
1. Xác nhận tên, props và mục đích component
2. Tạo folder: src/components/ComponentName/
3. Tạo file: index.tsx, styles.ts, types.ts
4. Tạo file test: ComponentName.test.tsx
5. Export từ barrel file

## Convention
- PascalCase cho component name
- Functional component + hooks
- Props interface riêng trong types.ts
